# オントロジー記述の有効性検証レポート

## 概要

オントロジーの因果ルールを構成する各プロパティ（クラス階層、因果方向、
期待値、条件、自然言語説明）が仮説検証の品質にどの程度寄与するかを
段階的に評価した。6段階のレベル（L0-L5）で7つの仮説を検証し、
5つの能力（仮説生成、SQL精度、判定精度、異常検出、施策提案）を
0-2点でスコアリングした。

### PoC環境

本評価は以下の環境で実施した: Python 3.12 / DuckDB 1.5 / dbt 1.11 / rdflib 7.6 / uv 0.11 / macOS (Apple Silicon)。
スコアリングはルールベース評価であり、LLMは使用していない。
なお、前段の仮説検証レポートではClaude Code（Claude Opus 4.6）をLLMとして使用し、
APIではなくClaude Codeのセッション内で直接コンテキストを読み取り仮説生成・SQL実行・検証を行った。

### データソースについて

本評価で使用しているデータは `src/ontoprobe/db/seeder.py` により生成された**合成データ**である。
オントロジーの因果ルールの検証を可能にするため、季節商品のQ4集中、VIPの高額購買、
キャンペーンによる注文数増加といったパターンが意図的に埋め込まれている
（`random.seed(42)` で再現性を確保）。
評価スコアはこの合成データの特性を前提としており、実データでは結果が異なる可能性がある。

## オントロジーのプロパティ解説

本プロジェクトのオントロジーは3つの層で構成されている。

### 第1層: 構造定義（ecommerce.ttl）

#### クラス階層（`owl:Class`, `rdfs:subClassOf`）

DBのカラム値にドメイン上の「意味」を与えるプロパティ。

```turtle
:Customer a owl:Class .
  :VIPCustomer rdfs:subClassOf :Customer .    # customer_segment = 'vip'
  :NewCustomer rdfs:subClassOf :Customer .    # customer_segment = 'new'

:Product a owl:Class .
  :SeasonalProduct rdfs:subClassOf :Product . # is_seasonal = true

:Campaign a owl:Class .
  :DiscountCampaign rdfs:subClassOf :Campaign .      # campaign_type = 'discount'
  :FreeShippingCampaign rdfs:subClassOf :Campaign .   # campaign_type = 'free_shipping'
```

`is_seasonal`というカラム名だけでは「何の季節性か」が不明。クラス定義により「需要が特定季節に集中する商品」という意味を機械可読にする。

#### メトリクスマッピング（`:measuredBy`）

オントロジーの概念とセマンティックレイヤー（dbtメトリクス）を接続する橋渡しプロパティ。

```turtle
:Revenue a :Metric ;
    :measuredBy "total_revenue" .        # ← dbtメトリクス名

:OrderVolume a :Metric ;
    :measuredBy "order_count" .
```

オントロジーで「Revenue」と言ったときに、具体的にどのメトリクス（`SUM(total_amount)` on `fct_orders`）で測定するかを定義する。これがないとオントロジーとデータベースが繋がらない。

#### 業務制約（`:hasConstraint`）

```turtle
:Order :hasConstraint :MinOrderValue .                  # total_amount > 0
:DiscountCampaign :hasConstraint :MaxDiscountPercent .   # discount_percent <= 50
```

データの妥当性チェックや異常値判定の基準を定義する。

### 第2層: 因果ルール（causal_rules.ttl）

仮説検証の核となる7つのプロパティ。

#### `:hasCause` — 原因

```turtle
:rule_discount_order_volume :hasCause :DiscountCampaign .
```

因果関係の原因側。クラス階層のインスタンスを指す。「何が」売上変動を引き起こすかを特定する起点。

#### `:hasEffect` — 結果（影響を受けるメトリクス）

```turtle
:rule_discount_order_volume :hasEffect :OrderVolume .
```

因果関係の結果側。Metricインスタンスを指す。`:measuredBy`を経由してdbtメトリクスに到達し、SQL生成が可能になる。

#### `:hasDirection` — 効果の方向

```turtle
:rule_discount_order_volume :hasDirection "increase" .
```

値は `"increase"` または `"decrease"`。方向がないと「関係がある」しか分からず、検証結果の判定（supported/contradicted）ができない。

#### `:hasExpectedMagnitude` — 期待される効果の大きさ

```turtle
:rule_discount_order_volume :hasExpectedMagnitude "15-30%" .
:rule_seasonal_revenue :hasExpectedMagnitude "2-3x" .
:rule_repeat_clv :hasExpectedMagnitude "positive correlation" .
```

**本検証で最も効果的と判明したプロパティ。** これがないと「増えた」としか言えない。これがあると「期待15-30%に対して+7%なので期待以下」「期待2-3倍に対して15倍なので異常」と定量的に判定できる。

記述のコツ:
- 「15-30%」のような**数値範囲**が最も有効
- 「positive correlation」のような曖昧な記述は判定精度が低い

#### `:hasCondition` — 適用条件

```turtle
:rule_seasonal_revenue :hasCondition "order_quarter = 4" .
:rule_discount_order_volume :hasCondition "discount_percent > 10" .
```

SQL生成時の`WHERE`句に直結する。「いつ・どんな条件で」ルールが適用されるかを限定することで検証精度が上がる。

記述のコツ:
- SQLで使える形式（`order_quarter = 4`）が望ましい
- 「冬に」より「order_quarter = 4」の方がSQL生成に直結

#### `:hasComparedTo` — 比較対象

```turtle
:rule_vip_higher_aov :hasComparedTo :NewCustomer .
```

「VIPのAOVが高い」だけでは「何と比べて？」が不明。比較ベースラインを明示することで、SQL生成（`WHERE segment IN ('vip','new')`）と判定基準（New比+40-60%）が明確になる。

#### `:hasDescription` — 自然言語説明

```turtle
:rule_seasonal_revenue :hasDescription
    "Seasonal products (winter fashion, gift sets) see 2-3x revenue increase in Q4" .
```

構造化プロパティだけでは表現しきれない背景知識を補完する。「冬物衣料・ギフト」のような情報がLLMの施策提案を具体化する。ただし**構造化プロパティが先、説明は補完**。

### 第3層: プロパティの接続関係

```
クラス階層           因果ルール              セマンティックレイヤー
─────────           ──────────              ──────────────────
:VIPCustomer ──hasCause──→ :CausalRule
                           │
                           ├──hasEffect──→ :AverageOrderValue ──measuredBy──→ "average_order_value"
                           │                                                         │
                           ├──hasDirection──→ "increase"                              ↓
                           ├──hasMagnitude──→ "40-60%"                     AVG(total_amount)
                           ├──hasCondition──→ (なし)                       on fct_orders
                           ├──hasComparedTo──→ :NewCustomer
                           └──hasDescription──→ "VIP customers are..."
```

オントロジー（ドメイン知識）→ メトリクスマッピング → セマンティックレイヤー → SQLという経路で、抽象的な因果知識が具体的なデータ検証に変換される。

## 4. 主要な発見

### 最大の効果: L2→L3 (期待値)

合計 **+21点** の改善。

### プロパティ別の効果分析

#### L0→L1: クラス階層 + メトリクスマッピング

- **主な効果:** 仮説の方向付け（「何を調べるべきか」のヒント）
- **限界:** 因果関係が不明なため「AとBに関係があるかも」レベルの仮説しか立てられない
- クラス階層がないと、`is_seasonal`カラムの意味すら推測に頼る

#### L1→L2: 因果方向（cause → effect + direction）

- **主な効果:** 仮説生成とSQL精度が大幅改善
- 「X→Yが増加する」と明示されることで、具体的で検証可能な仮説を立てられる
- 判定も「方向が合っているか」までは可能に
- **最もコストパフォーマンスの高いプロパティ**の候補

#### L2→L3: 期待値（magnitude）

- **主な効果:** 判定精度と異常検出が大幅改善
- 期待値があることで「+7%は期待の15-30%より低い」「+601%は異常」と判定可能
- **異常検出の鍵:** 期待値なしでは「多い/少ない」しか言えないが、期待値があれば「想定の何倍か」を定量評価
- 施策提案も「期待に達していない」→「改善が必要」と根拠付けが可能に

#### L3→L4: 条件・比較対象（condition, comparedTo）

- **主な効果:** 一部仮説でSQL精度と施策提案が改善
- `discount > 10%`の条件で検証範囲を限定し精度向上
- `comparedTo: NewCustomer`で比較ベースラインを明示
- **効果は仮説依存:** 条件が定義されている仮説でのみ改善（全仮説に効くわけではない）

#### L4→L5: 自然言語説明（description）

- **主な効果:** 施策提案の具体性が一部改善
- 「冬物衣料・ギフト」のような背景情報で、商品カテゴリ別の深堀りが可能に
- 「購買頻度ベースの分析に切り替え」のような代替アプローチを提案可能に
- **改善幅は最も小さい:** 構造化プロパティ（L1-L4）が充実していれば追加効果は限定的

## 5. 結論と推奨

### 効果的なオントロジー記述の最小構成

```
L3（因果方向 + 期待値）で全体スコアの 77% を達成
L5（フルオントロジー）では 93%
```

**最小有効構成は L3（クラス階層 + メトリクスマッピング + 因果方向 + 期待値）**

この構成で仮説検証に必要な5能力の大部分をカバーできる。

### プロパティの優先度

| 優先度 | プロパティ | 効果 |
|-------|-----------|------|
| **必須** | クラス階層 + メトリクスマッピング | 仮説の対象を定義する基盤 |
| **必須** | 因果方向（cause → effect + direction） | 検証可能な仮説を立てるために不可欠 |
| **必須** | 期待値（magnitude） | 異常検出と判定精度の鍵 |
| 推奨 | 条件・比較対象 | 検証精度向上（仮説依存） |
| 任意 | 自然言語説明 | 施策提案の補強（LLM利用時に有効） |

### プロパティの有効性の序列

```
必須 ──────────────────────────────────────────── 任意
│                                                  │
│  クラス階層     因果方向       期待値      条件      説明文
│  measuredBy     hasCause       magnitude   condition  description
│                 hasEffect                  comparedTo
│                 hasDirection
│                                                  │
│  「何を」       「どう」       「どれだけ」  「いつ」    「なぜ」
│  調べるか      変わるか       変わるか     適用か     そうなるか
│                                                  │
│  L1: 20%        L2: 47%        L3: 77%     L4: 86%    L5: 93%
```

L3（クラス階層 + マッピング + 因果方向 + 期待値）で全体の77%をカバーできる。残りの23%は条件・比較対象・説明文による補完。

### 記述のコツ

1. **期待値は具体的な数値範囲で記述する:** 「positive correlation」より「15-30%」の方が検証精度が高い（Repeat CLV仮説で実証）
2. **条件はSQLで使える形式で記述する:** 「冬に」より「order_quarter = 4」の方がSQL生成に直結
